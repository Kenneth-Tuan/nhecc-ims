## Phase 1: 探索與定義 (Discover & Define)

目標：定義出「誰」是「核心使用者」，以及他們「最痛」的問題。

1. 拆解使用者角色 (User Segmentation)15-75 歲跨度太大，不能當成同一種人，請強制拆成三組不同的 Persona (人物誌)：
   - A 組：數位原住民 (15-25 歲)：學生、對 UI 挑剔、沒耐心、習慣手機操作。
   - B 組：社會中堅 (26-55 歲)：家長或進修者、忙碌、講求效率、能接受複雜功能。
   - C 組：數位移民 (56-75 歲)：退休進修者、視力退化、對科技恐懼、害怕按錯。
2. 執行「極端用戶」訪談 (Key User Interview)不要訪談中間的 B 組（他們適應力最強），專攻 A 組和 C 組。

   - 問 C 組：「平常怎麼報名課程？用 Line 還是打電話？最怕按到什麼？」
   - 問 A 組：「最討厭學校系統哪一點？」

3. 定義「最小公倍數」痛點 (Common Pain Points)
   - 假設訪談結果是：「C 組怕字太小看不到」、「A 組討厭步驟太多」。
   - 產品核心原則 (Design Principle)：確立本產品必須是「極簡、大字體、防呆」。

## Phase 2: 設計與原型 (Design & Prototype)

目標：產出一個「75 歲阿嬤也會用」的 Flow。

1. User Flow 規劃：從 C 組 (長輩) 的視角出發

   - 如果 75 歲都能順利完成報名，那 15 歲的一定沒問題（這叫 Inclusive Design 包容性設計）。
   - 關鍵決策：
   - 登入機制：長輩記不住帳密，一定要做 「手機號碼 + 簡訊驗證碼 (OTP)」 或 「Line 第三方登入」。千萬不要做「帳號/密碼/確認密碼」的註冊表單。
   - 導覽列：不要用漢堡選單 (Hamburger Menu)，長輩不知道那是選單。使用底部的 Tab Bar (標籤列) 並搭配「文字標籤」。

2. Wireframe 繪製 (Nuxt UI 預想)

   - 您現在已經在用 Nuxt UI，請腦補  <UButton size="xl"> 。
   - 字體策略：預設字體要比一般系統大 2px (例如 body 預設 18px)。
   - 色彩對比：符合 WCAG AA 標準（Nuxt UI 預設配色通常有達標，但要檢查自定義顏色）。

3. 紙本/低保真驗證

   - 拿畫好的大按鈕草圖，去給身邊 60 歲以上的長輩看：「阿姨，如果要報名歌唱班，你會按哪裡？」

## Phase 3: 系統架構與開發 (Architecture & Develop)

目標：用 Nuxt 建構高包容性的 MVP。

1. 資料庫設計 (Schema Design)

   - 除了  Course ,  User  之外，還要考慮  FamilyAccount  (家庭帳號)。
   - 情境：很多時候是「媽媽幫小孩報名」或「孫子幫阿公報名」。Schema 要支援  User A  幫  User B  報名課程的關聯。

2. 前端實作重點 (Nuxt UI)

   - RWD 優先：這個年齡層跨度，90% 操作會在手機上。請先寫  <div class="md:hidden">  的手機版面。
   - 無障礙開發 (A11y)：
   - Nuxt UI 的組件大多支援，但您要注意  aria-label 。
   - 例如：「上一頁」按鈕如果是箭頭 icon，一定要加  aria-label="回到上一頁" ，讓視障者的螢幕報讀軟體能讀出來。

## Phase 4: 測試與發布 (Test & Deploy)

目標：確保不會有人因為「看不懂」而打電話來客訴。

1. 易用性測試 (Usability Testing)

   - 找一位 70 歲的測試者，請他完成「登入 -> 找課程 -> 報名」三個動作。
   - 觀察指標：他猶豫了幾秒？他在哪裡瞇起眼睛（代表字太小）？

總結：您的第一步行動清單

明天早上起床，您可以先做這三件事：

1. 列出 Persona：寫下「林阿嬤 (72 歲)」和「陳小弟 (16 歲)」兩個角色的能力值與恐懼點。
2. 決定「登入方式」：這會決定整個系統的門檻。建議研究 Firebase Auth 或 Supabase 的手機登入/Line 登入實作。
3. 畫出「報名流程圖」：只畫「報名」這一條線，並強迫自己把步驟縮減到 3 步以內（例如：選課 -> 輸入電話 -> 確認）。
