在產品開發（特別是軟體或企業系統）的情境下，**Key User**（關鍵使用者）指的是在該產品的目標使用群體中，**最具代表性、影響力或擁有深厚領域知識（Domain Knowledge）的使用者代表**。

簡單來說，他們是「使用者中的 VIP」或「超級使用者」，在開發過程中扮演著橋樑的角色，連結了「一般終端使用者」與「產品開發團隊/IT 部門」。

### Key User 的主要特徵與角色職責

1.  **需求定義與確認（Requirements Definition）**

    - Key User 通常非常熟悉現有的作業流程或業務痛點。開發初期，他們負責代表終端使用者提出具體需求，並與 PM 或開發者釐清「哪些功能是必要的」、「哪些流程是正確的」[1][7]。
    - 他們不一定懂程式，但他們最懂「現場是怎麼運作的」。

2.  **驗收測試（UAT - User Acceptance Testing）**

    - 在產品上線前，Key User 是進行 UAT 的核心人物。他們會實際操作系統，檢查功能是否符合業務邏輯，並決定是否放行（Go/No-Go）[7]。

3.  **種子教官（Trainer / Evangelist）**

    - 當產品上線後，Key User 通常會成為內部培訓的講師，負責教導其他一般使用者如何使用新系統，因為他們最了解新舊系統的差異與操作邏輯 [7]。

4.  **決策輔助與權衡**
    - 當開發資源有限，需要取捨功能（Trade-off）時，Key User 能提供「業務優先級」的建議，協助判斷哪些功能可以晚點做，哪些必須現在做 [1]。

### 與其他類似角色的差異

為了避免混淆，以下用表格比較 **Key User** 與其他常見角色：

| 角色                         | 重點關注 | 差異點                                                                                                              |
| :--------------------------- | :------- | :------------------------------------------------------------------------------------------------------------------ |
| **Target User** (目標客群)   | 市場區隔 | 這是指產品想賣給「誰」（例如：25-35 歲上班族），是一個**群體概念**，不是指特定的合作個人 [2]。                      |
| **End User** (終端使用者)    | 實際操作 | 最終真正使用系統的所有人。Key User 是 End User 的一部分，但 End User 不一定有權參與決策或測試。                     |
| **Product Manager** (PM)     | 產品成敗 | PM 負責產品的整體規劃與時程，需整合 Key User 的需求並轉化為規格，但 PM 不一定具備 Key User 的深層業務操作經驗 [1]。 |
| **Stakeholder** (利害關係人) | 專案利益 | 包含老闆、出資者、部門主管等。Key User 也是 Stakeholder 之一，但更專注於「操作層面」與「功能實用性」。              |

### 實際應用場景舉例（以您的物流系統背景為例）

假設您正在開發一個新的「空運提單管理系統」：

- **End User**：所有會用到系統的 50 位報關人員。
- **Key User**：其中 1-2 位資深報關員。他們知道哪種提單格式最容易出錯、海關查驗最常看哪個欄位。開發時，您會專門找這兩位開會確認欄位設計；測試時，也是由他們先測過沒問題，才推廣給其他 48 人使用。

### 總結

**Key User** 就是您在開發時最需要「抱大腿」的對象。搞定 Key User，通常就等於搞定了需求釐清與上線後的推廣；若忽視他們，開發出的產品往往會「不接地氣」，導致上線後充滿抱怨。
