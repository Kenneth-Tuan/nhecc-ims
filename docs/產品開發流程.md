1. 定義需求 (Requirement Definition)  
   輸入： Key User 的痛點  
   輸出： User Stories (使用者故事)  
   在這個階段，不要想「功能 (Feature)」，要想「價值 (Value)」。
   - 痛點：「我在田裡巡視時，還要拿紙筆抄數據，回家還要打進電腦，很累又容易抄錯。」
   - User Story 格式：
     作為一個 角色，我想要 行動，以便於 獲得的價值。
   - 範例：「作為一個農場管理員，我想要在手機上直接輸入巡檢數據，以便於省去重複輸入的時間並減少錯誤。」
2. 梳理流程 (Flow Charting)  
   輸入： User Stories  
   輸出： User Flow / Task Flow (流程圖)  
   這就是我們上一題討論的部分，把故事變成邏輯路徑。
   - 思考點：他在田裡可能沒有網路（離線模式？）、手可能有泥土（按鈕要大？）。
   - 產出：畫出流程圖（Login -> 選擇田區 -> 快速輸入 -> 暫存/上傳）。
3. 功能規格與資料規劃 (Spec & Schema)  
   輸入： User Flow  
   輸出： Functional Map (功能地圖) & DB Schema  
   這是工程師最擅長的環節，也是連接 UX 與 Code 的橋樑。
   - Functional Map：把 Flow 拆解成具體功能點。
   - 巡檢功能 -> GPS 定位、拍照上傳、數值驗證、離線儲存 (Local Storage)。
   - Schema Design：根據功能決定資料結構。
   - InspectionLog  (id, farm_id, user_id, image_url, location_lat, location_lng, created_at)。
4. 介面原型 (Prototyping)  
   輸入： Functional Map & User Flow  
   輸出： Wireframe (線框圖) -> Mockup (視覺稿)  
   先畫草圖，確認佈局，不要一開始就寫 Code。
   - Wireframe (低保真)：用筆紙或簡單工具畫框框。確認：「輸入框放這裡順手嗎？」、「提交按鈕顯眼嗎？」
   - 驗證 (Validation)：這一步最重要！ 拿著這張醜醜的草圖去找 Key User。
   - 問他：「如果是這個畫面，你會怎麼操作？」
   - 如果他卡住了，直接修圖，比修 Code 快 100 倍。
5. 開發與迭代 (Development & Iteration)  
   輸入： 已驗證的 Wireframe/Mockup  
   輸出： MVP (最小可行性產品)  
   現在才打開 VS Code / Cursor 開始寫 Nuxt。
   - MVP 思維：不要做「完美」的系統，先做「能解決最痛那個點」的最小功能。
   - 例如：先做數據輸入，報表分析功能之後再說。

1. 核心架構：雙鑽石模型 (Double Diamond)  
   這是由英國設計協會 (Design Council) 提出的標準架構 ，用來描述「從問題到解決方案」的過程。之所以叫「鑽石」，是因為它的思考過程是「發散 (Diverge) -> 收斂 (Converge)」的形狀：
   - 第一個鑽石（探索問題）：做對的事 (Do the right thing)
   - 發散 (Discover)：訪談 Key User，蒐集一堆痛點（這就是您說的「找 key user」）。
   - 收斂 (Define)：將痛點整理、篩選，寫成 User Story，確認「我們要解決哪一個核心問題」（這就是您說的「寫成 user story」）。
   - 第二個鑽石（設計解法）：把事做對 (Do the thing right)
   - 發散 (Develop)：畫 User Flow、Wireframe，發想各種可能的解法（這就是您說的「規劃接下來要做的事」）。
   - 收斂 (Deliver)：測試驗證，最後寫成 Code (MVP)，交付給使用者。
2. 實戰執行：精實 UX (Lean UX)  
   在雙鑽石的架構下，因為您是一人或小團隊，不可能跑完長達數月的設計流程。所以工程師通常會採用 Lean UX 的精神 ：
   - 核心精神：Think -> Make -> Check (思考 -> 製作 -> 驗證)
   - 特點：不寫厚重的規格書，而是把 User Story 快速轉化為 Flow 和原型，盡快讓程式碼上線去驗證假設。

不用拘泥於「標準名詞」，重點是**「雙鑽石」的節奏**：

1. 先發散找問題（不要預設解法）。
2. 收斂鎖定一個痛點（不要貪心）。
3. 再發散想流程（不要只用一種解法）。
4. 最後收斂寫成 Code（做出來）。
